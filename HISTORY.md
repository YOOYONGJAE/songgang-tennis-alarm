# 개발 과정 기록

코트 빈자리 알림 봇을 만들어온 전 과정을 순서대로 정리한 문서입니다.
직접 진행한 계정·설정 작업(봇 토큰, chat_id, Secrets 등록 등)은 **(직접)** 으로 표시했습니다.

---

## 1단계 · 데이터 통로 찾기 (분석)

1. 예약 사이트를 방문해 코트 예약가능 표기(`fit` 테이블)를 확인 → 정적 HTML엔 없고 JavaScript가 채운다는 걸 발견.
2. JS가 부르는 숨은 주소 `place_month_state_list` 를 찾아 직접 호출 → 응답의 `state_cd` 가 `10`=예약가능 / `20`=예약불가임을 파악.
3. 코트 = `place_code` 1~4 임을 확인.

## 2단계 · 기획

4. 자동화 그림을 그림: 주기적으로 조회 → 빈자리 생기면 텔레그램 알림 → 날짜 기간도 설정 가능.
5. 스택 결정: 실행은 GitHub Actions, 상태 저장은 Upstash, 알림·설정은 텔레그램, 무료를 위해 공개 리포.

## 3단계 · 코드 작성

6. `checker.py`(조회·명령·알림·상태저장) + 워크플로 + README 를 작성.

## 4단계 · 배포와 계정 셋업

7. **(직접)** 텔레그램 **@BotFather** 에게 `/newbot` → 봇 만들고 **봇 토큰** 받음.
8. **(직접)** 만든 봇에게 말을 걸고, `getUpdates` 주소로 접속해 내 **chat_id** 확보.
9. **(직접)** Upstash 가입 → DB 생성 → **REST URL·토큰** 확보.
10. GitHub 공개 리포 생성 + 코드 push (리포는 직접 생성, 파일은 로컬 git으로 업로드).
11. **(직접)** GitHub **Secrets 4개** 등록: 봇 토큰, chat_id, Upstash URL, Upstash 토큰.
12. **(직접)** Actions 탭에서 수동 실행(Run workflow)으로 첫 가동 확인.

## 5단계 · 실전에서 부딪힌 문제 해결

13. 시간(회차)까지 알리려 시간 단위 조회로 바꿨더니 요청이 폭증 → 서버가 **연결 거부(rate limit)** → 월 단위로 되돌림.
14. GitHub 자체 예약(cron)이 몇 시간째 제대로 안 뜨는 걸 확인 → 외부 트리거로 가기로 결정.
15. **(직접)** GitHub **Personal Access Token**(fine-grained, 이 리포의 Actions 읽기·쓰기) 발급.
16. **(직접)** cron-job.org 에서 10분마다 GitHub `workflow_dispatch` 를 호출하는 job 설정(URL·헤더·바디), 토큰 권한을 바로잡아 404→204, Inactive → **Active** 로 활성화.
17. GitHub 자체 예약(schedule) 제거 → 트리거를 cron-job.org 하나로 단일화.

## 6단계 · 기능·안정성 다듬기

18. 생존 신고 추가: 빈자리 없어도 12시간마다 **"확인 중"** 통보.
19. 확인 기간이 끝나면 **"다시 설정하세요"** 1회 안내.
20. `/상태` 를 종합 정보(설정·현재 빈자리·마지막 확인 시각)로 확장.
21. 예약 링크를 각 자리의 코트·날짜에 맞게 수정.
22. 있던 자리가 다 사라지면 **"빈자리 없음"** 통보 추가.
23. 자동 재시도 + 사람처럼 요청 간격(2~5초) + 실패 원인별(응답 지연 / 연결 거부 / …) 경고 문구.
24. 표현 정리: "감시" → "확인", 특정 시설명 제거해 일반화.

## 현재 상태

25. cron-job.org(10분)가 GitHub 을 깨우고 → 코드가 코트를 확인해 → 새 빈자리·소멸·상태를 텔레그램으로 알리는 구조가 **완전 자동으로** 동작.

---

## 한 줄 요약

숨은 데이터 통로를 찾아(1단계) → 조회 코드를 만들고(2~3단계) → 계정을 붙여 배포하고(4단계) → 요청량·타이머 문제를 실전에서 잡고(5단계) → 기능과 안정성을 다듬은(6단계) 여정.

## 구성 요소 한눈에

- **데이터 출처**: 예약 사이트의 상태 조회 엔드포인트(`place_month_state_list`)
- **실행 엔진**: GitHub Actions (`workflow_dispatch` 로만 구동)
- **주기 트리거**: cron-job.org (약 10분마다)
- **상태 저장**: Upstash Redis (`config` / `notified` / `tg_offset` / `last_heartbeat` / `last_run` / `expiry_notified`)
- **알림·설정 창구**: 텔레그램 봇 (소유자 chat_id 전용)
